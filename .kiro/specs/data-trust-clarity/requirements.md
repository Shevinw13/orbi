# Requirements Document

## Introduction

This specification covers a data trust and clarity improvement pass for the Orbi iOS travel app. The goal is to improve user confidence in displayed data by unifying the restaurant selection model, making pricing explainable and numeric, clarifying the origin of itinerary items and ratings, adding external source links for all locations, and enforcing UI density constraints so new elements remain minimal.

## Glossary

- **Trip_Setup_Flow**: The preferences overlay (PreferencesOverlay / DestinationFlowView) where users configure trip parameters before itinerary generation.
- **Itinerary_Engine**: The backend service (itinerary.py) that generates AI-powered day-by-day itineraries via OpenAI.
- **Itinerary_View**: The iOS views (TripResultView, ItineraryView, InlineDaySectionView) that render the generated itinerary.
- **Place_Card**: The PlaceCard SwiftUI component in RecommendationsView.swift that displays a single hotel or restaurant recommendation.
- **Place_Service**: The backend service (places.py) that fetches hotel and restaurant recommendations from Foursquare or OpenAI.
- **Recommendations_View**: The RecommendationsView SwiftUI component that displays hotel and restaurant lists.
- **Restaurant_Selector**: A new lightweight UI component in Trip_Setup_Flow allowing optional pre-selection of up to 3 restaurants.
- **Item_Origin_Label**: A small text label on each itinerary card indicating whether the item was user-selected or system-generated.
- **Price_Formatter**: Logic on both backend and iOS that converts raw price data into human-readable numeric estimates.
- **External_Link**: A small tappable element on itinerary and recommendation cards that opens a Google search or Google Maps URL for the location.
- **Rating_Display**: The UI element showing a numeric rating with its source attribution.

## Requirements

### Requirement 1: Restaurant Pre-Selection in Trip Setup

**User Story:** As a traveler, I want to optionally select up to 3 restaurants during trip setup, so that my preferred dining spots appear in the generated itinerary.

#### Acceptance Criteria

1. WHEN the user reaches the Trip_Setup_Flow, THE Restaurant_Selector SHALL display available restaurant recommendations for the selected destination as a horizontally scrollable list.
2. WHEN the user taps a restaurant in the Restaurant_Selector, THE Restaurant_Selector SHALL toggle that restaurant's selection state.
3. WHILE the user has selected 3 restaurants, THE Restaurant_Selector SHALL prevent selection of additional restaurants.
4. WHEN the user has selected 0 restaurants, THE Trip_Setup_Flow SHALL allow itinerary generation to proceed without restaurant pre-selections.
5. THE Restaurant_Selector SHALL display each restaurant's name, cuisine type, and formatted price range.

### Requirement 2: Inject Selected Restaurants into Itinerary

**User Story:** As a traveler, I want my pre-selected restaurants to appear in the generated itinerary at appropriate meal slots, so that my dining preferences are honored.

#### Acceptance Criteria

1. WHEN the user has pre-selected restaurants, THE Itinerary_Engine SHALL inject the selected restaurants into lunch or dinner slots across the itinerary days.
2. WHEN the user has not pre-selected any restaurants, THE Itinerary_Engine SHALL continue auto-generating restaurant recommendations for each day as the current behavior.
3. THE Itinerary_Engine SHALL ensure no duplicate restaurants appear in the itinerary — each selected restaurant SHALL appear at most once.
4. WHEN the number of pre-selected restaurants is fewer than the number of itinerary days, THE Itinerary_Engine SHALL auto-generate restaurants for the remaining days.

### Requirement 3: Itinerary Item Origin Labels

**User Story:** As a traveler, I want to see whether each itinerary item was chosen by me or suggested by the system, so that I understand the source of each recommendation.

#### Acceptance Criteria

1. WHEN an itinerary item was pre-selected by the user, THE Itinerary_View SHALL display the Item_Origin_Label "Selected by you" on that item's card.
2. WHEN an itinerary item was auto-generated by the Itinerary_Engine, THE Itinerary_View SHALL display the Item_Origin_Label "Suggested" on that item's card.
3. THE Item_Origin_Label SHALL render as small secondary text that does not increase the card's height or displace existing content.

### Requirement 4: Hotel Pricing as Numeric Estimates

**User Story:** As a traveler, I want to see hotel prices as numeric per-night averages instead of dollar-sign tiers, so that I can understand the actual cost.

#### Acceptance Criteria

1. THE Place_Card SHALL display hotel pricing in the format "$XXX / night avg" where XXX is a numeric dollar amount.
2. THE Place_Card SHALL display the label "Estimated" adjacent to the hotel price.
3. WHEN the Place_Service returns price_range_min and price_range_max values, THE Price_Formatter SHALL compute the average of those values for display.
4. WHEN the Place_Service does not return numeric price range data, THE Price_Formatter SHALL map the price tier to a fallback estimate: "$" maps to $80, "$$" maps to $150, "$$$" maps to $250.
5. THE Place_Card SHALL NOT display standalone dollar-sign tier strings (e.g. "$", "$$", "$$$") as the sole price representation for hotels.

### Requirement 5: Restaurant Pricing as Numeric Ranges

**User Story:** As a traveler, I want to see restaurant prices as per-person numeric ranges instead of dollar-sign tiers, so that I can budget my meals.

#### Acceptance Criteria

1. THE Place_Card SHALL display restaurant pricing in the format "$XX–$XX per person" where XX values are numeric dollar amounts.
2. THE Place_Card SHALL display the label "Estimated" adjacent to the restaurant price.
3. WHEN the Place_Service returns price_range_min and price_range_max values, THE Price_Formatter SHALL use those values directly for display.
4. WHEN the Place_Service does not return numeric price range data, THE Price_Formatter SHALL map the price tier to a fallback range: "$" maps to $10–$20, "$$" maps to $20–$40, "$$$" maps to $40–$70.
5. THE Place_Card SHALL NOT display standalone dollar-sign tier strings (e.g. "$", "$$", "$$$") as the sole price representation for restaurants.

### Requirement 6: Itinerary Restaurant Pricing Clarity

**User Story:** As a traveler, I want restaurant prices within the itinerary day view to also show numeric ranges, so that pricing is consistent across the app.

#### Acceptance Criteria

1. THE Itinerary_View SHALL display each day's restaurant price in the format "$XX–$XX per person" instead of the raw priceLevel string.
2. WHEN the restaurant's priceLevel is a dollar-sign tier, THE Price_Formatter SHALL map the tier to a numeric range using the same mapping as Requirement 5.
3. THE Itinerary_View SHALL display the label "Estimated" adjacent to the restaurant price.

### Requirement 7: External Source Linking for All Locations

**User Story:** As a traveler, I want to open any location in Google Maps or Google Search, so that I can verify details and get directions.

#### Acceptance Criteria

1. THE Place_Card SHALL display a small "View" link or icon for each hotel and restaurant recommendation.
2. THE Itinerary_View SHALL display a small "View" link or icon for each activity slot and restaurant in the itinerary.
3. WHEN the user taps the External_Link on any item, THE app SHALL open a URL in the format "https://www.google.com/maps/search/?api=1&query={Place_Name}+{City}" using the system browser or in-app Safari.
4. THE External_Link SHALL render as a small tappable element that does not increase the card's height or displace existing content.

### Requirement 8: Rating Source Attribution

**User Story:** As a traveler, I want to know where ratings come from, so that I can trust the displayed scores.

#### Acceptance Criteria

1. WHEN a rating is displayed, THE Rating_Display SHALL show the numeric rating followed by the source in parentheses, e.g. "4.5 (Google)" or "4.5 (Foursquare)".
2. WHERE the Place_Service provides a review_count, THE Rating_Display SHALL optionally show the review count, e.g. "(1,200 reviews)".
3. WHEN the Place_Service does not provide a rating or the rating value is 0, THE Rating_Display SHALL hide the rating element entirely instead of showing a zero or empty rating.
4. THE Itinerary_View SHALL apply the same rating source attribution format for restaurant ratings within day sections.

### Requirement 9: Backend Price Range Data Enrichment

**User Story:** As a developer, I want the backend to provide numeric price range estimates alongside tier strings, so that the iOS app can display explainable pricing.

#### Acceptance Criteria

1. WHEN the Place_Service generates restaurant recommendations via OpenAI, THE Place_Service SHALL include price_range_min and price_range_max fields in each result based on the price tier: "$" yields min=10, max=20; "$$" yields min=20, max=40; "$$$" yields min=40, max=70.
2. WHEN the Place_Service generates hotel recommendations via OpenAI, THE Place_Service SHALL include price_range_min and price_range_max fields in each result based on the price tier: "$" yields min=60, max=100; "$$" yields min=120, max=180; "$$$" yields min=200, max=300.
3. WHEN the Place_Service fetches results from Foursquare, THE Place_Service SHALL apply the same tier-to-range mapping if Foursquare does not provide numeric price data.

### Requirement 10: UI Density Constraints

**User Story:** As a traveler, I want the app to remain clean and uncluttered, so that new trust elements do not overwhelm the interface.

#### Acceptance Criteria

1. THE Item_Origin_Label SHALL use the caption2 font size and textTertiary color from DesignTokens.
2. THE "Estimated" price label SHALL use the caption2 font size and textTertiary color from DesignTokens.
3. THE External_Link SHALL use the caption font size and accentCyan color from DesignTokens.
4. THE Rating_Display source attribution SHALL use the caption2 font size and textTertiary color from DesignTokens.
5. THE Restaurant_Selector SHALL present as a horizontal scroll within the existing Trip_Setup_Flow layout without adding new full-screen sections.
